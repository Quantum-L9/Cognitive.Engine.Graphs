# Domain Spec Authoring

The domain spec is the source of truth for all matching behavior (Contract 12).
No matching logic is hardcoded in the engine.

## Layout ‚Äî this matters

`DomainPackLoader` resolves **exactly one** path:

```
domains/<domain_id>/spec.yaml
```

The flat `domains/<name>_domain_spec.yaml` files are **not loadable**. They are
reference drafts. `DomainPackLoader.list_domains()` returns only directories
containing a `spec.yaml` ‚Äî today that is `plasticos` alone. Putting a new spec at
the flat path produces a `DomainNotFoundError` at load time, not a validation
error, so it fails late and confusingly.

Symlinked `spec.yaml` files are rejected by the loader.

## Steps

1. **Confirm the domain ID is unused**: no `domains/<domain_id>/` directory yet.
2. **Read the schema**: `engine/config/schema.py` defines every spec type.
3. **Read a working example**: `domains/plasticos/spec.yaml` is the only
   loader-validated spec in the repo.
4. **Create `domains/<domain_id>/spec.yaml`** with these sections:
   - `domain` ‚Äî `id`, `name`, `version` (quoted, see Versioning below)
   - `ontology` ‚Äî node types, edge types, property declarations
   - `match_entities` ‚Äî source and target entity definitions
   - `query_schema` ‚Äî input parameter definitions
   - `traversal` ‚Äî path patterns, hops bounded (max 10)
   - `gates` ‚Äî filters using the 10 gate types
   - `scoring` ‚Äî ranking dimensions using the 13 computation types
   - `sync` ‚Äî entity ingestion endpoints
   - `compliance` ‚Äî PII fields and handling mode, if applicable
5. **Validate**:
   ```bash
   python tools/validate_domain.py domains/<domain_id>/spec.yaml --strict
   ```
   Add `--json` for CI-parseable output.
6. **Confirm the loader sees it**:
   ```bash
   python -c "from engine.config.loader import DomainPackLoader; \
     print(DomainPackLoader('domains').list_domains())"
   ```
7. **Add a test** in `tests/unit/test_config_loader.py` asserting the spec loads
   and required sections are present.
8. **Run `make agent-check`**.

## Versioning

Format is `{major}.{minor}.{patch}-{stage}` (`docs/contracts/DOMAIN_SPEC_VERSIONING.md`).
The stage records how the spec was produced, so pick it by origin, not by taste:

| Origin | Version |
|--------|---------|
| Auto-generated by the ENRICH CRM field scan | `0.1.0-seed` |
| Hand-authored, complete, reviewed | `1.0.0` |

A hand-written spec starting at `0.1.0-seed` falsely claims the enrichment
pipeline produced it. Stages advance `seed ‚Üí discovered ‚Üí inferred ‚Üí proposed
‚Üí reviewed ‚Üí production`; each transition may add columns but never drop them.

## Naming

YAML keys are the Python field names ‚Äî there are no Pydantic aliases (NAME-001
is a banned pattern). Gate type and scoring computation identifiers are flatcase
(`temporalrange`, `geodecay`); structural keys are snake_case (`match_entities`).

## Gate types (10)

`range`, `threshold`, `boolean`, `composite`, `enummap`, `exclusion`,
`selfrange`, `freshness`, `temporalrange`, `traversal`

## Scoring computations (13)

`geodecay`, `lognormalized`, `communitymatch`, `inverselinear`,
`candidateproperty`, `weightedrate`, `pricealignment`, `temporalproximity`,
`customcypher`, `traversalalias`, `kge`, `variantdiscovery`, `ensembleconfidence`

## Validation checklist

- [ ] Spec lives at `domains/<domain_id>/spec.yaml`
- [ ] `list_domains()` includes the new domain
- [ ] Edge `source`/`target` types reference declared node types
- [ ] Gate field references match declared node properties
- [ ] Scoring weight overrides sum ‚â§ 1.0 (asserted at boot in `engine/boot.py`)
- [ ] Traversal hops bounded (max 10)
- [ ] No prohibited compliance fields referenced (Contract 10 ‚Äî fails at compile time)
- [ ] `python tools/validate_domain.py ... --strict` exits 0
